# Go Fiber Boilerplate

A production-ready boilerplate for building REST APIs with **Fiber**, **GORM**, **PostgreSQL**, **JWT authentication**, SQL migrations, structured logging, and Scalar API documentation.

This template is intentionally generic. It includes authentication, user profile endpoints, a neutral `Resource` CRUD example, ready-to-use SMTP password reset email, optional integration skeletons, Docker support, and a clear layered architecture that can be reused for future API projects.

## Features

- **Fiber Web Framework** - Fast, minimalist HTTP framework for Go.
- **JWT Authentication** - Access and refresh token flow with strict `Authorization: Bearer <token>` headers.
- **Scalar API Reference** - Interactive API documentation at `/docs` in non-production environments.
- **OpenAPI Generation** - Swagger spec generated by `swaggo/swag`.
- **DTO Layer** - Request and response contracts with self-validation.
- **Struct-Based Handlers** - Handlers receive service interfaces through constructors.
- **Service Interfaces** - Business logic behind explicit interfaces for easier testing and replacement.
- **SQL Migration-First Schema** - Runtime schema changes are driven by SQL files, not GORM AutoMigrate.
- **GORM ORM** - Database abstraction with PostgreSQL runtime support and SQLite-friendly test utilities.
- **Structured JSON Logging** - Request IDs, module names, daily log rotation, and redacted request/response bodies.
- **Middleware Stack** - Request ID, request context, panic recovery, CORS, Helmet, rate limiting, compression, access logs, and centralized error handling.
- **Optional Redis Cache** - Redis-backed cache and rate-limit storage with no-op fallback when Redis is not configured.
- **SMTP Email** - Ready-to-use password reset email with no-op fallback when SMTP is not configured.
- **Optional Integration Skeletons** - Storage interface with a no-op implementation.
- **Docker Support** - Production and development Dockerfiles, plus Docker Compose with Air hot reload.
- **Testing Infrastructure** - Test database setup, fixtures, and assertion helpers.

## Quick Start

### Prerequisites

Choose one setup.

**Option A: Docker + Hot Reload**

- Docker Desktop or Docker Engine
- Docker Compose
- No local Go or PostgreSQL installation required

**Option B: Local Development**

- Go 1.25 or newer
- PostgreSQL 12+
- Make

### Installation

1. Clone the repository.

```bash
git clone https://github.com/yourusername/go-fiber-boilerplate.git
cd go-fiber-boilerplate
```

2. Install dependencies.

```bash
make install-deps
```

3. Create environment file.

```bash
cp .env.example .env
```

4. Change required environment values.

```text
JWT_SECRET=your-project-specific-secret-at-least-32-characters
DB_NAME=your_database_name
CORS_ALLOWED_ORIGINS=http://localhost:3000,http://localhost:4000
```

5. Start the development environment.

```bash
make docker-dev
```

This starts PostgreSQL and the API with hot reload. The API listens on:

```text
http://localhost:4000
```

Useful URLs:

```text
Health:  http://localhost:4000/health
Docs:    http://localhost:4000/docs
Spec:    http://localhost:4000/swagger.json
```

View logs:

```bash
make docker-dev-logs
```

Stop development containers:

```bash
make docker-dev-down
```

```go
//	@title			Your API Name

//	@description	Your API description.
//	@host			localhost:4000
//	@BasePath		/api
```

Update these files:

- `.env.example`
- `README.md`
- `API_GUIDELINE.md` if your project changes template conventions
- Docker Compose service/container names if needed

### Step 3: Keep or Remove the Sample Resource

The template includes a generic `Resource` CRUD example:

- `internal/models/resource.go`
- `internal/dto/resource_dto.go`
- `internal/services/resource_service.go`
- `internal/handlers/resource_handler.go`
- `/api/resources` routes
- `resources` SQL migration table and seed data

Keep it as a reference implementation, rename it into your first real domain entity, or remove it after your own feature is added.

## Adding Your First Feature

Follow this order for new entities.

### 1. Add SQL Migration

Create a numbered migration in `assets/migrations`.

```sql
CREATE TABLE IF NOT EXISTS products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(120) NOT NULL,
    description TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_products_name ON products(name);
```

### 2. Add Model

```go
package models

import "time"

type Product struct {
	ID          uint      `gorm:"primaryKey" json:"id"`
	Name        string    `gorm:"type:varchar(120);not null" json:"name"`
	Description *string   `gorm:"type:text" json:"description,omitempty"`
	CreatedAt   time.Time `json:"created_at"`
	UpdatedAt   time.Time `json:"updated_at"`
}

func (Product) TableName() string {
	return "products"
}
```

### 3. Add DTOs

```go
type CreateProductRequest struct {
	Name        string  `json:"name" validate:"required,min=2,max=120"`
	Description *string `json:"description" validate:"omitempty,max=2000"`
}

func (r *CreateProductRequest) Validate() error {
	return validate.Struct(r)
}
```

### 4. Add Service

```go
type ProductService interface {
	CreateProduct(req *dto.CreateProductRequest) (*models.Product, error)
}

type productService struct {
	db *gorm.DB
}

func NewProductService(db *gorm.DB) ProductService {
	return &productService{db: db}
}
```

### 5. Add Handler

```go
type Product struct {
	productService services.ProductService
}

func NewProduct(productService services.ProductService) *Product {
	return &Product{productService: productService}
}
```

Handler responsibilities:

- Parse request body/query/path.
- Call `req.Validate()`.
- Read authenticated user context if needed.
- Call service.
- Map service errors to HTTP status.
- Return response via `pkg/utils` helpers.

### 6. Wire Routes

Initialize the service and handler once in `internal/routes/routes.go`, then register routes under `/api`.

### 7. Generate Docs and Verify

```bash
make swagger
make fmt
make vet
make test
make build
```

## Common Development Tasks

### Add Environment Variable

1. Add a field to `config.Config`.
2. Read it in `LoadConfig()`.
3. Add validation if required.
4. Add it to `.env.example`.
5. Document it in this README if it is user-facing.

### Add Migration

1. Create `NNN_description.sql` in `assets/migrations`.
2. Keep it PostgreSQL-compatible.
3. Add indexes for foreign keys and common filters.
4. Update the matching GORM model.
5. Run `make migrate` locally.

### Add Middleware

1. Add middleware to `internal/middleware`.
2. Keep domain-specific authorization behind reusable helpers such as `RequireRoles`.
3. Register global middleware in `cmd/api/main.go`.
4. Register route-specific middleware in `internal/routes/routes.go`.

### Write Tests

Use `internal/testutil`:

- `SetupTestDB(t)`
- `CleanupTestDB(db)`
- `CreateUserFixture`
- `CreateResourceFixture`
- assertion and JSON response helpers

## Authentication

Authentication uses JWT access and refresh tokens.

Login response:

```json
{
  "status": 200,
  "message": "Login successful",
  "data": {
    "token": "access-token",
    "refresh_token": "refresh-token",
    "expires_in": 900,
    "role": "user"
  }
}
```

Protected requests must use:

```http
Authorization: Bearer <access-token>
```

The auth middleware stores these values in `fiber.Ctx` locals:

- `user_id`
- `email`
- `role`

Use helpers:

```go
userID, err := middleware.GetUserIDFromContext(c)
email := middleware.GetEmailFromContext(c)
```

Role checks:

```go
group.Use(middleware.AuthMiddleware())
group.Use(middleware.RequireRoles("admin"))
```

## Database Migrations

Runtime schema is migration-first. SQL files are embedded into the binary through `assets/embed.go`.

Migration files:

```text
assets/migrations/001_initial_schema.sql
assets/migrations/002_add_indexes.sql
```

Seed files:

```text
assets/migrations/seeds/001_admin_user.sql
assets/migrations/seeds/002_sample_resources.sql
```

Development reset:

```bash
make migrate-fresh
make seed
```

Do not use GORM AutoMigrate for runtime schema. Test utilities may use AutoMigrate for temporary in-memory databases.

## Logging

Use module loggers:

```go
utils.Log("App").Info("Starting server")
utils.LogCtx(c.UserContext(), "Resource").Error("Create failed", "error", err)
```

Logging features:

- JSON logs to stdout and daily files under `logs/`.
- Request ID propagation.
- Access logs with method, path, status, latency, IP, user ID, and optional redacted JSON bodies.
- Sensitive keys such as password, token, authorization, secret, and API key are redacted.

## Project Structure Details

### `cmd/api/main.go`

Initializes logger, loads config, connects to the database, runs migrations, creates Fiber, initializes cache and limiter storage, registers middleware, wires routes, and handles graceful shutdown.

### `internal/routes/routes.go`

Composition root. Services are created once and injected into handlers.

### `internal/handlers`

HTTP layer. Handlers are struct-based and should not create services directly.

### `internal/services`

Business logic layer. Services expose interfaces and should not depend on Fiber.

### `internal/dto`

Request/response contracts and validation.

### `internal/models`

GORM entities that match SQL migrations.

### `internal/middleware`

Auth, optional auth, role checks, rate limiting, request context, access logging, and error handling.

### `internal/cache`

Optional Redis cache wrapper. Disabled safely when Redis is not configured or unreachable.

### `pkg/mailer`

Generic mailer interface and SMTP client used by the password reset email flow.

### `pkg/utils`

Response helpers, password hashing, structured logger, redaction, and miscellaneous reusable helpers.

### `pkg/jwt`

JWT token creation, validation, and Bearer header extraction.

## AI Assistant Notes

AI agents working in this repository may:

- Read files.
- Edit code and docs.
- Run `make fmt`, `make vet`, `make test`, `make build`, and `make swagger`.

AI agents must not:

- Run `make run`.
- Run `go run ./cmd/api`.
- Run `make dev` or `air`.
- Run `make docker-dev`, `make docker-up`, or `docker compose up`.
- Start any long-running server process.

## Production Checklist

- Set a strong `JWT_SECRET`.
- Restrict CORS origins.
- Enable database SSL when required by infrastructure.
- Review log body settings and redaction keys.
- Run migrations before serving traffic.
- Generate and commit updated OpenAPI docs.
- Verify `make test`, `make vet`, and `make build`.
- Use real implementations for optional integrations that your product depends on.
